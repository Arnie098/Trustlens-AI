# TrustLensAI — Mobile (Capacitor)

Capacitor shell that wraps the existing TrustLens web app (`../src`) for Android,
adds a camera/OCR **Scan** flow, share/clipboard verify paths, and an opt-in,
calm **Floating Assist** bubble.

- **Design (source of truth for product rules):**
  [`../docs/superpowers/specs/2026-07-17-mobile-capacitor-scanner-overlay-design.md`](../docs/superpowers/specs/2026-07-17-mobile-capacitor-scanner-overlay-design.md)
- **Implementation plan:**
  [`../docs/superpowers/plans/2026-07-17-mobile-capacitor-scanner-overlay.md`](../docs/superpowers/plans/2026-07-17-mobile-capacitor-scanner-overlay.md)

The web source in `../src` is the source of truth for the UI. This `mobile/`
package owns only the native project, plugins, and the sync tooling.
**Never hand-edit `www/`** — it is regenerated from the web build.

## Environment (device builds)

Values are baked into `www/` **at web-build time**. Copy `.env.example` to the
**repo root** as `.env` (or export in your shell) before syncing:

```bash
VITE_DB_PROVIDER=supabase
VITE_SUPABASE_URL=https://YOUR_PROJECT.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=YOUR_ANON_KEY
VITE_API_BASE_URL=https://YOUR_DEPLOYED_TRUSTLENS_HOST
```

- SQLite is **local-dev only** — device builds must use hosted Supabase.
- `VITE_API_BASE_URL` is **required on device**: the bundled `www/` cannot
  resolve a relative `/api/analyze` against a remote host without it.
- **Never** embed Perplexity or server secrets in the APK. Analyze stays on the
  deployed host.

## Build & run (from the repo root)

```bash
# 1. Build the web app into mobile/www and cap sync
npm run mobile:sync:cap

# 2. Open Android Studio and run on device/emulator
cd mobile && npm run open:android
```

Convenience: `npm run mobile:android` does the sync + opens Android Studio in one
step. `npm run mobile:sync` builds/copies without running `cap sync`.

### Re-adding the native project

`mobile/android/` is generated by `npx cap add android`. If it is missing (fresh
clone that gitignores it), run from `mobile/`:

```bash
npm install
npm run add:android   # npx cap add android
```

## Deep-link scheme

The app registers the custom scheme `ai.trustlens.app` (Capacitor default) plus a
readable `trustlens://` scheme (added in the manifest task). Deep-link contract
into the verify flow:

```
trustlens://verify?tab=scan|url|text|image
trustlens://verify?tab=text&prefill=<text>&source=clipboard|share|overlay|camera
```

Web/query equivalent used inside the WebView: `/verify?tab=scan&source=overlay`.

## Android permissions used

| Permission | Why | When requested |
| --- | --- | --- |
| `INTERNET` | Reach Supabase + analyze API | Always (install-time) |
| `CAMERA` | Scan a post with the camera | On first Scan → camera |
| `READ_MEDIA_IMAGES` (33+) / `READ_EXTERNAL_STORAGE` (≤32) | Import a screenshot | On first gallery pick |
| `SYSTEM_ALERT_WINDOW` | Floating Assist bubble over other apps | Only when user enables assist |
| `FOREGROUND_SERVICE` (+ `FOREGROUND_SERVICE_SPECIAL_USE`) | Keep the assist bubble alive | Only while assist is on |
| `POST_NOTIFICATIONS` (33+) | Neutral "assist is on" foreground notification | Only while assist is on |

Overlay + capture permissions are **optional** — the app works fully without
them. Floating Assist is **default off** and never re-enables itself.

## Performance QA checklist (design §18.7)

Run on **mid-range** Android (≈4–6 GB RAM), production build only:

- [ ] **Cold start ≤ 3.0 s** to interactive shell (warm ≤ 1.5 s), median of 10.
- [ ] **Tab navigation ≤ 200 ms** perceived; no multi-second white screens.
- [ ] **Scan → OCR → analyze:** UI stays tappable; memory does not climb
      unbounded across **10 consecutive runs** (revoke object URLs, recycle bitmaps).
- [ ] **Assist on, idle 30 min** while using TikTok/IG: TrustLens CPU ≈ 0%,
      no meaningful battery drain, no phone warmth (Battery Historian).
- [ ] **Side-by-side scroll test:** feed with assist off vs half-hidden on —
      a reviewer cannot tell which recording has the bubble from smoothness alone.
- [ ] **Expand/collapse bubble 50×:** no leaked windows, no growing RAM.

### Anti-annoyance QA (design §6.3, 3+ testers, 10 min IG/TikTok each)

- [ ] Floating assist **default off**; never force-enabled after install.
- [ ] Bubble tiny (40–48 dp), edge-docked, **auto half-hide ~2.5 s**.
- [ ] **No** unsolicited popups / panels / notification spam while scrolling.
- [ ] **No** clipboard polling — clipboard read only on explicit action.
- [ ] Hide / turn off reachable in **≤ 2 taps** (long-press menu).
- [ ] Testers agree it is "not distracting" and would keep it enabled a week.

### Profiling steps

1. Android Studio **Profiler** → CPU/Memory while assist idle over TikTok.
2. `adb shell dumpsys batterystats` / **Battery Historian** for the idle-drain run.
3. Screen-record two 30 s TikTok scrolls (assist off vs on) → compare frame drops.

## Native code map

Custom Android code beyond the Capacitor default lives under `android/app/src/main/java/ai/trustlens/app/`:

| Area | Path |
| --- | --- |
| Deep links, permission hooks | `MainActivity` |
| Floating assist service | `overlay/AssistBubbleService.kt` |
| Bubble view (drag, half-hide, long-press) | `overlay/AssistBubbleView.kt` |
| Mini action panel (≤ 35% height) | `overlay/AssistPanelView.kt` |
| JS ↔ native bridge for assist toggle | `overlay/AssistPlugin.kt` |
| Still-image OCR (ML Kit) | `ocr/OcrHelper.kt` |

See [`native/README.md`](native/README.md) for the overlay architecture and the
perf rules those files must honor.
